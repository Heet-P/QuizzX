# QuizzX Migration Audit

Source of truth for the QuizzX (Vite/React + Express) → QuizzX_V2 (Next.js 16) migration.
Generated by reading every file under `QuizzX/client/src` and `QuizzX/server/src` (plus root-level
docs, schema, and config) on 2026-07-21. `QuizzX` is treated as read-only reference throughout.

Use the checkboxes in each table during Phase 6 to confirm 1:1 behavior in `QuizzX_V2`.

---

## 0. Read this first — discrepancies vs. the migration brief

The migration brief assumes QuizzX currently runs on **Supabase** (Postgres + Storage). That is
**not what the code shows**. Findings, verified by direct search of both `client/src` and
`server/src`:

1. **No `@supabase/supabase-js` anywhere.** Not in `client/package.json`, not in
   `server/package.json`, not `require`d/`import`ed in any source file. The server talks to
   Postgres exclusively through a raw `pg.Pool` (`server/src/config/db.js`), already pointed at
   **Neon** in every real config (`README.md`, `server/.env.example` use `DB_HOST=your-project.neon.tech`).
   The only "Supabase" references left are stale comments (`db.js:4,28`: *"Force IPv4 for all
   connections (Render/Supabase IPv6 connectivity issue)"*, *"35 connections (Supabase free
   supports ~60)"*) — vestiges of an earlier setup that was already migrated off Supabase before
   this audit. **There is no Supabase client, no RLS policy, and no Supabase Realtime subscription
   to reimplement.** Phase 5.2's "flag Supabase-specific calls" step will find nothing to flag.
2. **No existing file storage system (Supabase Storage or otherwise).** Teacher/admin quiz uploads
   (`.docx`/`.txt`/`.md`) are parsed in-memory via `multer.memoryStorage()` + `mammoth`, converted
   to a `questions` JSONB array, and the original file bytes are discarded — never written to disk
   or any bucket (`AdminController.uploadQuiz`, `AdminController.uploadAnswerKey`). The team photos
   (`Heet Parikh.jpg`, etc.) are static assets checked into `client/public/`, not user uploads.
   **There is nothing to migrate off of storage-wise.** Introducing Cloudflare R2 in Phase 5 is
   *adding* a new capability (e.g. persisting the original uploaded quiz file, or future avatar
   uploads), not replacing a working integration. Confirm with the user what, if anything, should
   actually be stored in R2 before building it — see the open question at the bottom of this file.
3. **`supabase_schema.sql`** at the repo root is a Postgres DDL script that happens to work against
   either Supabase or Neon (it's just SQL) — it is not evidence of a Supabase SDK integration.
   The canonical, current, idempotent schema is `server/migrations/001_initial_schema.sql`
   (`CREATE TABLE IF NOT EXISTS`, safe to re-run). `server/schema.sql` is explicitly marked
   `-- REFERENCE SCHEMA — DO NOT RUN AGAINST PRODUCTION` (it `DROP`s tables first) and is for
   spinning up a brand-new environment only. All three files describe the same 14-table shape;
   `001_initial_schema.sql` is the one to `prisma db pull` against (or hand-convert to a Prisma
   schema) — see Section 5.
4. **`webapp_review.md`** (root, dated 2026-05-22) is a **stale, historical** review — several of
   its "current issues" no longer exist in the code read for this audit:
   - Legacy `/api/auth` routes / `AuthController.js` / `authRoutes.js` it flags as dead — **do not
     exist** in `server/src/routes/` or `server/src/controllers/` today.
   - `server/src/models/` (dead ORM-style layer) it flags — **does not exist** today.
   - `client/src/services/quizService.js` (unused API wrapper) it flags — **does not exist** today.
   - Client-side `seededShuffle` duplication it flags — **already removed**; `QuizPage.jsx` now
     explicitly comments *"Server already shuffles questions and options with a user+quiz seed.
     No client-side shuffle needed."*
   - `VITE_ADMIN_EMAILS`-based admin UI gating it flags as a security hole — **already fixed**;
     `Layout.jsx` now calls `GET /users/me` and gates on `role` returned from the DB. Note
     `VITE_ADMIN_EMAILS` is still present in `client/.env` but appears unused by any source file
     (see Section 9) — likely leftover.
   Treat `webapp_review.md` as background color on product intent, not as a description of current
   behavior. Everything in this audit was verified against the actual source, not the review doc.
5. **Three backend/frontend features exist with zero real caller** (confirmed via repo-wide grep):
   - `GameController` / `gameRoutes.js` / `game_sessions` table (`POST /game/sessions`,
     `PATCH /game/sessions/:id/score`, `POST /game/sessions/:id/complete`) — no page or component
     calls any `/game/...` endpoint. It's a redundant shadow of `SubmissionController`'s scoring
     path (it upserts into the same `submissions` table and pushes the same leaderboard update)
     with no capability the main quiz flow doesn't already provide.
   - `client/src/components/AdminQuizControls.jsx` — emits `admin:question:show` /
     `admin:question:close` / `admin:quiz:end` over the quiz Socket.IO channel. The server side
     (`socketService.js`) fully implements listeners for these events, but `AdminQuizControls` is
     never imported by any page. Its one capability (host paces question reveal live) is already
     delivered a different way by the Live Lobby feature (`RoomController` + `LiveLobbyPage`,
     REST + polling, not sockets).
   - `GET /leaderboard/stream` (SSE, `LeaderboardController.streamLeaderboard` +
     `utils/leaderboardBroadcast.js`) — fully built server-side (including a Redis pub/sub fan-out
     path) and already pinged on every submission (`lb.broadcast(quizId, {type:'new_submission',...})`),
     but **no client file ever opens an `EventSource` against it**. `LeaderboardPage.jsx`'s live
     updates come exclusively from `useQuizSocket` (Socket.IO). Confirmed by grepping for
     `EventSource` across `client/src` — the only hit is `ProctorPage.jsx`.
   **Decision (user, 2026-07-21):** drop `GameController`/`game_sessions` and `AdminQuizControls`
   entirely — consistent with the user's plan to run a separate "keep only essential features"
   audit after this migration. Do not port either. The leaderboard SSE endpoint is **not** dropped —
   see Section 7: since Socket.IO itself is being dropped (also decided 2026-07-21), this
   already-built-but-unconsumed SSE endpoint becomes the actual mechanism for live leaderboard
   updates in QuizzX_V2, so it needs a new client-side consumer rather than being deleted.

## 4. Backend API endpoints

Base mount: every router is mounted twice in `app.js` — under `/api/v1/<resource>` (canonical)
and `/api/<resource>` (back-compat alias). The old frontend's `VITE_API_URL` always included
`/api`, i.e. it only ever called the alias path, never `/api/v1/`.
**Decision (user, 2026-07-21): drop `/api/v1` entirely** — QuizzX_V2 serves everything at
`/api/<resource>` via `app/api/<resource>/.../route.ts`, matching what the app actually calls
today and Next.js's own `app/api` convention. Tables below list paths relative to `/api`.

Global middleware order (`app.js`): raw-token injection for SSE routes (see Section 7) →
`clerkMiddleware()` → CORS (explicit allowlist from `ALLOWED_ORIGINS`) → `express.json()` /
`urlencoded()` → `helmet()` → `morgan` → `GET /health` (DB ping, no auth, no `/api` prefix) →
`apiLimiter` (1000 req/15min/IP) on all `/api` → routers → global error handler (JSON
`{success:false,message,error?}`, `error` field only in dev).

## 7. Real-time / WebSocket / SSE

**Decision (user, 2026-07-21): drop Socket.IO entirely, consolidate on SSE/polling.** Rationale:
Socket.IO needs a long-lived Node process, which typical Next.js hosting (Vercel-style serverless)
doesn't provide; SSE and polling both work fine from ordinary Next.js Route Handlers anywhere.
This is a genuine behavior change from the old app (documented here rather than silently applied),
but it's smaller than it first looks once you account for what each feature actually used:

**What the old app actually used, per feature** (this matters — it's not "Socket.IO for
everything"):
- **Leaderboard live updates** (`LeaderboardPage.jsx`) — 100% Socket.IO (`useQuizSocket`). **But**
  the SSE equivalent (`GET /leaderboard/stream`) was already fully built server-side and already
  gets pinged on every submission — it just never had a frontend consumer (Section 0.5). **Action:
  build a new `useLeaderboardStream` hook that opens an `EventSource` against `/api/leaderboard/stream`**,
  mirroring the pattern already proven in `ProctorPage.jsx` (reconnect-with-backoff on `onerror`).
  The REST fallback path (`GET /leaderboard` on filter change) that already exists stays as-is.
- **Live lobby** (`LiveLobbyPage.jsx`, `PresenterPage.jsx`) — **already 100% REST + polling** (3s/2s
  intervals) on the client side. Socket.IO was only used *outbound*, from `RoomController.answerQuestion`/
  `endRoom` pushing to the `/leaderboard` page's socket room — it never fed the lobby/presenter
  pages themselves. **No change needed here**; these two pages port over unchanged.
- **Proctor stream** (`ProctorPage.jsx`) — already SSE-only. Ports over unchanged.
- **Presence count** (`presence:update` Socket.IO event, shown nowhere in the current UI — grep
  found no page reading `participantCount` from `useQuizSocket`) — **dropped, no replacement**.
  Confirmed cosmetic/unused: `useQuizSocket` exposes it but no page consumes it.
- **Per-question live reveal** (`question:active`/`question:closed`, driven by the orphaned
  `AdminQuizControls`) — **dropped along with `AdminQuizControls`** (Section 0.5); the Live Lobby's
  polling-based `advance` flow is the shipped way to do paced question reveal and needs no change.

**Net effect:** porting Sections "Live lobby" and "Proctor" needs zero real-time architecture
changes (they were never on Socket.IO to begin with). The only real work is replacing
`useQuizSocket` with an SSE-based hook for the leaderboard page, using server infrastructure that
already exists (`utils/leaderboardBroadcast.js`'s in-process registry + optional Redis pub/sub
fan-out, `app.js`'s `?__session=` → `Authorization` header rewrite for `EventSource`, which can't
set custom headers — port that rewrite into the new SSE route handler too).

**Not ported:** `server/src/services/socketService.js` in its entirety, `@socket.io/redis-adapter`
+ `socket.io` + `socket.io-client` dependencies, `client/src/hooks/useQuizSocket.js` (replaced),
`client/src/components/AdminQuizControls.jsx` (dropped, Section 0.5), the two
`// 🧪 TEMP DEBUG` leftover debug-logging blocks that lived in those files.

- [ ] New SSE-based leaderboard hook matches old Socket.IO consumer shape closely enough that
      `LeaderboardPage` needs minimal changes (rows, connection status, last-update timestamp)
- [ ] `/leaderboard/stream` SSE route ports the `?__session=` → `Authorization` header rewrite
- [ ] Live lobby (`/live`, `/live/:code`, `/live/:code/display`) ports with no real-time-layer changes
- [ ] Proctor SSE stream + reconnect behavior ports with no changes
- [ ] Confirmed dropped, not silently missing: presence count, per-question live-reveal admin controls

## 8. `QuizPage.jsx` state machine (the most complex single piece of frontend logic)

Local component state across ~15 `useState`s plus 3 `useRef`s. Order of screens:

1. **Loading** → fetch quiz (`GET /quizzes/:id`); 403+`completed:true` → short-circuit to
   "already completed" state (skips rules/quiz entirely).
2. **Challenge banner** (optional) — if `?challenge=<clerkId>` present, fetches challenger's
   score via `GET /users/challenge-info`.
3. **Rules screen** (`RulesScreen`) — shown until `onStart`. If `settings.accessCode` set, PIN
   modal (`PinGateModal`, calls `POST /quizzes/:id/verify-pin`) must pass first.
4. **Start**: `POST /submissions/start` (nonce) → request fullscreen (`useQuizCheating.startQuiz`,
   iOS/unsupported → scroll-lock fallback instead of failing).
5. **In progress**: draft answers persisted to `localStorage` (`quiz_progress_{id}`) on every
   change, additionally auto-saved server-side every 10s (`POST /submissions/draft`) via a
   ref-diffed interval (avoids re-registering the interval on every keystroke). Tab-switch/copy/
   fullscreen-exit detection via `useQuizCheating` logs events (`POST /submissions/events`) and,
   depending on `settings.tabSwitch` (`auto_submit` | `three_strikes` | `disabled`), can trigger
   auto-submit. `settings.answerLock` (`changeable` | `lock_on_select` | `lock_on_next`) controls
   whether an answered question can be changed. `settings.navigation` (`free` | `forward_only` |
   `sequential_locked`) controls whether "Previous" is available / prior questions are locked.
   Question nav palette (desktop sidebar + mobile toggle) shows answered/flagged/unreachable state.
6. **Submit**: `ConfirmModal` confirmation → `POST /submissions` → on success shows a full-screen
   success overlay with buttons to Results or Leaderboard, plus a separate level-up celebration
   overlay if `leveledUp`; achievement toasts fire staggered 800ms apart; XP toast fires
   immediately. On error, a full-screen retry overlay.
7. **Practice mode** (`timer==='none' && tabSwitch==='disabled'`) is a materially different path:
   answers are present in the fetched question objects, selection immediately shows correct/wrong
   feedback + explanation inline, no fullscreen/nonce/auto-submit logic applies.
8. Legacy `settings.mode` (`timed_full`/`timed_per_question`/`practice`) is translated to the
   current settings shape client-side (`s` `useMemo`) for backward compatibility with quizzes
   created before the settings schema had explicit `timer`/`visibility`/`navigation` fields — the
   **same translation also exists server-side** in `AdminController.uploadQuiz` for new uploads.
   Keep both translations consistent if porting.

- [ ] All settings axes reproduced: `timer` (global/per_question/none), `visibility` (single/all),
      `navigation` (free/forward_only/sequential_locked), `answerLock` (changeable/lock_on_select/
      lock_on_next), `tabSwitch` (auto_submit/three_strikes/disabled), `copyProtection`,
      `shuffleQuestions`, `shuffleOptions`, `pointsPerCorrect`, `negativeMarking`, `accessCode` (PIN),
      `poolSize`/`showCount`, `quiz_mode`/`max_team_size`, `startAt`/`endAt` window
- [ ] localStorage draft + server draft auto-save both work and don't fight each other
- [ ] Legacy `mode` → settings translation matches server-side translation exactly
- [ ] Practice mode fully bypasses lockdown/nonce/fullscreen logic

## 9. Cross-cutting logic that must stay in sync (candidates for real shared code in Next.js)

Since QuizzX_V2 unifies frontend and backend in one TypeScript project, the following
currently-duplicated logic (kept in sync by hand today, across two separate JS codebases) should
become genuinely shared modules (e.g. `lib/quiz-settings.ts`, `lib/answer-matching.ts`) rather than
copy-pasted again:

- **Answer matching / letter resolution** — `server/src/utils/mapAnswerLetter.js` used by
  `SubmissionController`, `RoomController`, `AdminService.getQuizAnalytics`, `AIController`. The
  client's own practice-mode correctness check (`QuestionCard`/`QuizPage`: `option === q.answer`)
  is a simpler *exact-match-only* check — it does **not** run the same letter/positional/normalize
  logic the server uses for real grading. Decide whether practice-mode feedback should be upgraded
  to use the same matching function for consistency, or intentionally left simple (it's low-stakes
  since no score is at risk in practice mode) — flag this choice rather than silently picking one.
- **Quiz settings shape + defaults** — defined ad hoc in `AdminController.uploadQuiz` (server
  defaults + legacy `mode` translation), `QuizUploader.jsx` `PRESETS` (client authoring defaults),
  `QuizPage.jsx` `s` `useMemo` (client runtime defaults + legacy `mode` translation again). Good
  candidate for one `QuizSettings` TypeScript type + one default/normalize function used
  everywhere.
- **Seeded shuffle** — `QuizController.js` `seededShuffle` (server-only now, client trusts server
  order — see Section 0.4). Single implementation already; just carry it over once.
- **CSV building** — near-identical `Content-Type`/`Content-Disposition`/line-join pattern appears
  in `AdminController.getIntegrityReport` and `LeaderboardController.exportLeaderboard`. Worth one
  small shared helper.
- **`normalize`/`letterOf` inline closures** — redefined per-function in `SubmissionController`,
  `RoomController`, `AdminService` instead of reusing `mapAnswerLetter`'s internal helpers. Same
  root logic, three copies — consolidate.
